# 乡村应急广播与预警信息联动小程序 —— 产品需求文档（PRD）

## 1. 项目背景与目标

### 1.1 背景
以"乡村暴雨夜应急预警"为典型场景。乡村地区地形复杂、信息触达慢、应急响应链条长。
当水位、雨量超过安全阈值时，需要第一时间将预警以"广播 + 小程序推送"的方式触达村民，
并驱动网格员完成现场核查与处置，形成可追溯的闭环。

### 1.2 目标
- 实现**预警自动触发**：依据水位/雨量等传感器数据，按规则自动生成预警与广播内容。
- 实现**多渠道触达**：文本转语音（TTS）应急广播 + 小程序消息推送。
- 实现**处置闭环**：触发 → 派发 → 确认 → 处置 → 归档 全流程工作流。
- 实现**双角色协同**：网格员（处置者，兼任后台管理与复核）与村民（接收者）各自的小程序视图。
- 系统**轻量化**：后端 Go 单体可独立部署，低运维成本，适配乡镇有限算力。

## 2. 用户角色

系统仅设**两个角色**：村民与网格员。网格员**兼任后台管理职责**（预警复核、规则与设备维护、
工作流监控），不再单设独立管理员。

| 角色 | 描述 | 核心诉求 |
| --- | --- | --- |
| 村民 | 普通村民用户 | 及时收到预警、查看避险指引、查看广播内容、上报隐患 |
| 网格员 | 片区责任人（兼后台管理员） | 接收派发任务、确认、现场处置、上传反馈；**校验复核系统预警、维护规则与设备、监控工作流** |

> **预警发布原则**：预警以**系统自动触发并发布为主**（基于传感器数据 + 规则引擎）。
> 人工**不是**主要发布途径，网格员的核心职责是**对系统预警进行校验、复核、确认或撤销**；
> 仅在传感器失效、人工巡查发现险情等**兜底场景**下，网格员才手动补发预警。

## 3. 功能需求

### 3.1 后台管理与预警复核（网格员 —— 以校验复核为主）
- F1 **系统预警校验/复核**：网格员对系统自动生成的预警进行复核，可"确认发布 / 修改文案 / 撤销误报"。
- F2 人工兜底发布：仅在传感器失效或人工发现险情时，手动补发预警（生成 TTS 广播）。
- F3 预警模板管理：内置暴雨/洪水/地质灾害模板，支持变量占位（如 `{村名}` `{水位}`）。
- F4 文本转语音（TTS）：将预警文本转为语音文件，供应急广播终端/小程序播放。
- F5 预警规则管理：配置传感器阈值规则（如水位 > 3.5m 触发橙色预警）。
- F6 设备管理：登记水位/雨量传感器（**规模约 100 台**），查看在线状态与最新读数。
- F7 工作流监控：查看每条预警的流转状态、处置时效、归档情况。
- F8 数据趋势与统计：查看设备**每日最高/平均水位**趋势曲线（基于日聚合数据）。

### 3.2 自动触发引擎（系统主发布途径）
- F9 传感器数据接入：约 100 台设备定时上报读数（平峰 1 次/分钟，暴雨期可提频）。
- F10 规则匹配：读数命中阈值规则后，**自动生成并发布预警**，进入工作流。
- F11 校验模式（可配置）：
  - 直发模式（默认高级别）：命中即发布，事后网格员复核；
  - 待审模式（可对低级别开启）：先生成 `pending_review` 待网格员确认后再发布。
- F12 防抖与去重：同一设备同级别预警在冷却期内不重复生成。
- F13 自动升级/降级：水位持续升高自动提升预警级别。
- F14 时序数据治理：原始读数热库保留近 30 天，每日降采样为日聚合数据长期保留，
  过期原始数据归档至云存储后清理（对用户透明，保障系统长期轻量运行）。

### 3.3 网格员端小程序
- F15 任务推送接收：收到派发任务的订阅消息推送。
- F16 任务确认：查看任务详情，点击"确认接收"，进入处置中状态。
- F17 现场处置反馈：填写处置说明，上传现场照片/视频（多张）。
- F18 任务上报完成：提交后任务进入待归档。
- F19 我的任务列表：按状态筛选（待确认/处置中/已完成）。

### 3.4 村民端小程序
- F20 预警接收与查看：首页展示本区域当前生效预警与级别。
- F21 广播内容播放：在线播放/收听 TTS 语音广播。
- F22 避险指引：查看对应灾种的避险知识与就近避难点。
- F23 隐患上报：村民可上报现场隐患（文字 + 图片 + 定位），生成线索。
- F24 历史预警查询：查看本村历史预警记录。

## 4. 核心业务流程（工作流）

```
   传感器读数（系统主途径）        人工兜底（网格员，少量）
        │ 规则引擎匹配                      │
        ▼                                  ▼
   [系统自动生成预警] ──(可选待审)──► [人工校验/复核]
        │ 直发 / 复核通过                   │ 确认/修改/撤销
        ▼◄─────────────────────────────────┘
   [触发 Triggered]
        │ 生成广播文本/TTS
   [派发 Dispatched] ── 推送给责任网格员 + 推送村民
        │ 网格员确认
   [确认 Confirmed]
        │ 现场处置 + 上传照片/视频
   [处置 Handled]
        │ 网格员审核
   [归档 Archived]
```

状态机：`(pending_review) → triggered → dispatched → confirmed → handled → archived`
说明：高级别预警默认跳过 `pending_review` 直发；低级别可配置经人工复核。
异常分支：误报由网格员 `canceled`；`dispatched` 超时未确认 → 自动重派/升级提醒。

## 5. 预警级别定义

| 级别 | 颜色 | 含义 | 示例阈值（水位） |
| --- | --- | --- | --- |
| Ⅳ 一般 | 蓝色 | 关注 | > 2.5m |
| Ⅲ 较重 | 黄色 | 准备 | > 3.0m |
| Ⅱ 严重 | 橙色 | 转移准备 | > 3.5m |
| Ⅰ 特别严重 | 红色 | 立即转移 | > 4.0m |

## 6. 非功能需求
- 容量：支持约 **100 台传感器**，平峰 1 次/分钟（≈1.67 QPS 写入），暴雨提频峰值 ≈10 QPS。
- 性能：设备上报响应 < 100ms（落库即返回，慢操作异步）；预警触发到推送送达 < 5s。
- 数据治理：原始读数热库保留 30 天，每日降采样为日聚合长期保留，过期数据归档云存储后清理；
  趋势/统计查询走日聚合表，保证长期运行性能稳定。
- 可用性：核心发布与推送链路具备失败重试；离线消息可补推；异步任务失败持久化补偿。
- 可追溯：全流程操作留痕，处置证据（照片/视频）持久化存储。
- 轻量化：后端单二进制（Gin + dbr，PostgreSQL 分区表 + 进程内调度/队列），Docker 一键部署。
- 安全：角色权限隔离，接口鉴权（JWT），设备上报签名校验。

## 7. 验收指标（MVP）
- 能通过**模拟水位脚本**自动触发一条橙色预警并完成全流程归档（系统主途径）。
- 能用 `sim_water_level.py --devices 100` 并发压测，验证后端在 100 设备规模下的吞吐与成功率
  （更大规模可用 `sim_water_level_1000.py`）。
- 网格员可对系统预警进行校验/复核（确认/修改/撤销），并能人工兜底补发一条预警。
- 系统预警与人工预警均能生成可播放的 TTS 语音。
- 日聚合任务能正确生成每日最高/平均水位，趋势接口可查询。
- 网格员端能收到推送、确认任务并上传至少 1 张现场照片。
- 村民端能查看预警、播放广播、提交一条隐患上报。

## 8. 范围与里程碑
- MVP（第一阶段）：自动预警 + 人工复核、工作流、双端核心、TTS、推送、分区表 + 日聚合治理。
- 第二阶段：MQTT 设备接入、数据大盘、统计报表、多乡镇租户。
- 升级路径：设备规模 ≥5000 或秒级采集时，readings 迁移至 TimescaleDB。
- 暂不做：第三方广播硬件深度对接（仅预留接口）、复杂 GIS 分析。
